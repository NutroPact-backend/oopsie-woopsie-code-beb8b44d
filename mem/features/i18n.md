# Three layers — use the right one

## 1. Hardcoded UI string the developer writes → `<T>` or `useAutoT`
Easiest, zero setup. Auto-translates via AI on first render, caches forever in `content_translations` table (entityType `ui_string`).
```tsx
import { T, useAutoT } from "@/lib/useContentT";
<T>Add to Cart</T>
<T>Saved for later</T>
const label = useAutoT("Continue shopping");
```
Use this for EVERY new visible English string. Length cap 200 chars.

## 2. Curated key with reviewed copy → `useT(key)` + i18n.dict.ts
For high-traffic strings where you want hand-reviewed translations.
```tsx
import { useT } from "@/lib/i18n";
const t = useT(); {t("cta.add_to_cart")}
```
Add key to `en` in `src/lib/i18n.dict.ts`; regen with `scripts/gen-i18n-dict.ts`.
Reuse existing: `cta.add_to_cart`, `cta.buy_now`, `cta.checkout`, `nav.*`, `footer.*`, `video.views`.

## 3. Admin-authored DB content (headings, banners, custom pages) → `useContentTBatch`
For ANY content the admin types into the dashboard (video section headings, page block titles, banner copy, etc.). Auto-translates first viewer, caches per (entity, field, locale).
```tsx
import { useContentTBatch } from "@/lib/useContentT";
const ct = useContentTBatch([
  { entityType: "video_section", entityId: s.id, field: "heading", source: s.heading },
]);
ct("video_section", s.id, "heading"); // returns translated or source
```
**Rule for future admin-built pages/sections**: always wrap admin DB strings with `useContentTBatch` on render. Pick a stable `entityType` (e.g. `banner`, `page_block`, `faq_item`).

# Infrastructure
- Table: `public.content_translations` (entity_type, entity_id, field, locale, source_hash, translated)
- Server fn: `getContentTranslations` in `src/lib/content-translations.functions.ts` — batch read + auto-AI fill via `getAIConfig()` (gemini-2.5-flash by default)
- Fallback chain: cache → source string (no UI block ever)
- Cache invalidates automatically when source string changes (source_hash mismatch → re-translate on next view)
